# Per-environment `requirements.txt` reference

A `requirements.txt` produced by `pip-compile` is specific to one (Python version, OS, implementation)
combination. {doc}`/explanation/cross-environment` covers the why; this page is the short reference.

## What "specific" means

The combination is determined by the interpreter that ran `pip-compile`:

| Axis | Examples that produce different lockfiles |
|------|-------------------------------------------|
| Python version | 3.9, 3.10, 3.11, 3.12, 3.13 |
| Free-threading | 3.13t versus 3.13 |
| Implementation | CPython, PyPy, GraalPy, RustPython |
| Operating system | Linux, macOS, Windows |
| CPU architecture | x86_64, aarch64, arm64 |
| C library | glibc versus musl on Linux |

Two compiles run on different combinations from the table can produce different output even when the
input is identical.

## Naming convention

A common scheme:

```text
requirements/
    linux-py3.10.txt
    linux-py3.13.txt
    linux-py3.13t.txt
    macos-arm64-py3.13.txt
    windows-py3.13.txt
```

Pick a naming scheme that captures the axes that vary in your project. If everyone is on Linux+CPython,
only Python version varies. If you support macOS too, add the OS.

## CI matrix

GitHub Actions example, compile-and-test matrix:

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, macos-latest, windows-latest]
    python: ["3.10", "3.13"]

steps:
  - uses: actions/setup-python@v5
    with:
      python-version: ${{ matrix.python }}
  - run: pip install pip-tools
  - run: pip-compile -o requirements/${{ matrix.os }}-py${{ matrix.python }}.txt
  - run: pip-sync requirements/${{ matrix.os }}-py${{ matrix.python }}.txt
  - run: pytest
```

The compile and the sync use the same lockfile name, derived from the matrix axes.

## Header line

The first line of the lockfile names the Python version that produced it:

```text
# This file is autogenerated by pip-compile with Python 3.13
```

Diagnostic only. `pip` will install the file under any Python; the header lets a reviewer or CI logs
spot a mismatch.

## When one file is enough

A single lockfile covers your project when all of these hold:

- All developers and CI run the same OS.
- All developers and CI run the same Python version.
- No package in your dependency tree has platform-conditional dependencies that would diverge.

Most teams meet the first two. The third is harder to verify without compiling on multiple platforms and
diffing. The conservative answer is "compile per platform you ship to".

## What does not help

- Markers in the source (`requests; sys_platform == "darwin"`) propagate to the output but only describe
  top-level conditional dependencies. They cannot fix transitive divergence.
- Hand-editing the lockfile to add per-platform pins. The format does not support multi-version lines.
- Running `pip-compile` once and copying the output to other platforms. The file may install on those
  platforms, but the resolution is not guaranteed correct.
