# Why `requirements.txt` is per-environment

A `requirements.txt` produced by `pip-compile` is specific to the Python interpreter and operating system
that produced it. Use it on a different combination, and you will see install failures or, worse, silent
divergence. Knowing why turns this from "weird footgun" into "expected design".

## What changes between environments

Three axes of variation matter:

- **Python version**: 3.9 versus 3.13, free-threaded versus standard. A package that requires
  `tomli; python_version < "3.11"` resolves to `tomli` only on 3.10 and below; on 3.11+ it resolves to
  nothing because Python ships `tomllib`.
- **Implementation**: CPython versus PyPy versus GraalPy. Some packages ship CPython-only wheels; others
  fall back to a sdist that requires a compiler.
- **Operating system and CPU architecture**: Linux x86_64 versus macOS arm64 versus Windows x86_64.
  Native-extension packages ship platform-specific wheels with different transitive dependencies.

A constraint declared with PEP 508 markers, for example `requests; sys_platform == "darwin"`, evaluates at
resolve time. The platform that ran `pip-compile` is the platform the marker saw.

## Why one `requirements.txt` cannot cover everything

`pip-compile` resolves to *concrete* versions, then writes them to the file. The resolver picks one
version per package, the version that fits the environment that ran the resolver. The file format does
not encode "on Linux, use 1.2; on Windows, use 1.3" per line.

The format stays small and predictable. Any `pip` can install it. The trade-off: one `requirements.txt`
covers one environment.

## What "the same `requirements.txt` across environments" breaks

Three failure modes show up:

1. **Version not available**. The file pins `numpy==2.0.0` because Linux x86_64 had a wheel. Your Apple
   Silicon laptop has no matching wheel and no compiler. `pip install -r` fails.
2. **Wrong transitive set**. The Linux compile resolved a package that has different transitive deps on
   Windows. The file is missing the Windows-only package, or contains an extra package that breaks
   Windows imports.
3. **Marker mismatch**. The file contains `tomli==2.0.1`. On Python 3.11 that line installs anyway
   (markers in the file are weaker than markers in the input), and `pip` then complains about a
   redundant dependency.

The first two are loud. The third is silent, which is worse.

## Producing one file per environment

Treat `requirements.txt` as a per-(Python, OS) artifact. Compile each combination separately, in the
target environment. Name the files so the combination is obvious:

```text
requirements/
    linux-py3.10.txt
    linux-py3.13.txt
    macos-py3.13.txt
    windows-py3.13.txt
```

CI runs the matching `pip-sync` for each platform. Local development uses whichever file matches the
local interpreter.

A common shortcut: if all your developers and your CI run the same OS and Python (say, Linux 3.13), one
file is enough. The shortcut breaks the day someone runs the project on a different machine. Plan for
multiplicity from the start; it costs nothing on day one and saves a sprint on day fifty.

## Why the header includes the Python version

The first line of every generated file looks like:

```text
# This file is autogenerated by pip-compile with Python 3.13
```

That string is informational. `pip` will install the file under a different Python. The string exists so
reviewers and CI logs can spot a mismatch. When the header says `3.10` and CI runs `3.13`, that is the
place to look first.

The command line in the header is normalised: one-off flags like `--upgrade` are stripped, defaults are
omitted, secrets in URLs are redacted. Re-running the header command reproduces the same file.
{doc}`/how-to/custom-compile-command` covers overriding it for wrapper scripts.

## When PEP 508 markers help (and when they don't)

Markers in `requirements.in` propagate. A line like `pywin32; sys_platform == "win32"` resolves to
`pywin32==X.Y` only on Windows; on Linux the resolver skips it. The output file carries the marker
forward. That works for top-level conditional dependencies.

Markers do not help for transitive packages whose existence depends on the environment. The resolver only
sees the transitive graph the *current* environment exposes. A Windows-only transitive dep is not there
during a Linux compile, and no marker rescues it.

The right answer when this matters is "compile per environment". The wrong answer is "guess the markers";
guessing produces brittle files that work until they don't.
